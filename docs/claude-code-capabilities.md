# Claude Code / Ultracode Capability Inventory

Sources checked: 2026-07-13. This file summarizes official Claude Code documentation as the current comparison target for Conductor. It is not proof that Conductor matches these capabilities.

Local CLI observation on this machine: `claude --version` reported `2.1.206 (Claude Code)`.

Availability caveat: official Claude Code behavior is version- and surface-dependent. Current official model/CLI documentation states that Claude Code 2.1.203+ accepts `--effort ultracode`, which applies xhigh reasoning plus dynamic workflow orchestration when workflows are available. Local 2.1.205 runs accepted the flag with `--model sonnet` and resolved the main model to `claude-sonnet-5`; strict evidence still validates the stream, main/helper models, terminal state, and cost because requested flags alone are insufficient provenance.

## Current Conclusion

Conductor now covers a deterministic automatic front door, explicit model workflow generation/execution, a safe Claude-style saved-workflow compiler with deterministic returned-result consolidation and common static `parallel()`/`phase()` compatibility, reusable policy-bounded agent profiles with immutable operator-selected progressive Skill trees, isolated or restricted Codex tools, revisioned external memory, bounded adaptive teams, isolated writes, durable local-terminal recovery, merge intents, and exactly-once source transactions. It also provides runtime-hard effort/token/session ceilings, structured and grouped fan-out, detached runs, content-free notifications, zero-replay local completion recovery, bounded graph replacement, same-session adaptive planning, restart-persistent local routines, strict Sonnet Ultracode adapters, one-command verified staged delivery, hash-bound scoring, and strong preflight gates. Claude Code with Ultracode remains ahead on effort-triggered orchestration without a separate command, arbitrary JavaScript functions/control flow/transforms and nested fan-out, provider-native exact mid-call continuation, interactive teammate-session takeover and split panes, cloud-persistent scheduling/triggers, cloud-task notifications, and multi-surface integration. Conductor is mechanically stronger on optional operator plan control, static script rejection, content-free provenance, explicit caps, and hash-bound local recovery, but broad/universal superiority remains unproved and cost comparison remains blocked by unavailable Codex dollar telemetry.

Release `0.152.0` superseded the broad native Skill-dependency gap for operator-pinned unauthenticated HTTPS `streamable_http` dependencies with exact tool allowlists. Releases `0.153.0`-`0.157.0` added separately approved authentication, hash-only continuation, hybrid lexical memory, a credential-isolating broker, and sandboxed selected Skill-local stdio MCP. Releases `0.158.0`-`0.165.0` tightened verified-delivery accounting, prompt/effort efficiency, cutoff provenance, and explicit crash-recoverable one-command staged application. Release `0.166.0` added the non-evaluating Claude-style saved-workflow compiler, inline Codex output-schema support, and strict JSON-pointer packet handoff; `0.167.0` added deterministic terminal result collection, drift reconciliation, and explicit context rendering; `0.168.0` adds common static `parallel()`/`phase()` compatibility and hash-bound opaque prompt items. OAuth login/stored OAuth, literal headers, arbitrary ambient/package-runner stdio commands, automatic app/plugin activation, semantic endpoint isolation, embedding/synonym semantics, complete JavaScript workflow semantics, and matched continuation/memory quality evidence remain open.

The next evidence milestone is a repeated blinded, task-diverse suite with independently scored outcomes and budget cohorts whose enforcement can be compared honestly, not a broader claim from the current narrow samples.
