# Codex Conductor

Codex Conductor is a clean-room orchestration project for secure Codex coding workflows.

It has two layers and a combined release bundle:

- `codex-conductor/`: the installable text-only Skill.
- `conductor_runtime/`: the dependency-free companion workflow runtime for automated, resumable, safety-gated workflows.
- `dist/codex-conductor-bundle.zip`: the Skill, runtime zip application, integrity manifest, installer, workflows, benchmarks, and docs in one artifact.
- `dist/codex-conductor-marketplace.zip`: a native local Codex marketplace containing the Skill and the same embedded runtime.

## What It Does

- Plans complex coding tasks.
- Creates visible workflow artifacts for broad or resumable work.
- May be selected as a prompt-only skill for goal-driven coding and repository work when implicit invocation is enabled.
- Uses optional Codex-native subagents when explicitly requested or clearly useful; the runtime can grant a separate bounded read-only native envelope.
- Requires explicit chat-visible approval before risky actions (a prompt-level gate the model follows, not a technical enforcement mechanism; Codex's own sandbox separately enforces filesystem/network boundaries).
- Requires verification evidence before success claims.
- Provides an optional runtime that can validate, dry-run, resume, and execute declarative workflow specs.
- Exports dependency-free JSON Schemas for public workflow, reusable agent profile with immutable progressive Skill-tree bindings, agent memory, content-free Codex progress and single-step terminal evidence, resumable agent-team state, operator inbox, chat, and private transcript, interruption, turn, locally terminal turn, provider-completed turn, hook-input, merge-intent/transaction/recovery, and quality-retry envelopes, auto-routing receipt, auto-topology and read-only parity campaigns, background-run, content-free desktop-notification, background-model-goal, background-routine-supervisor, benchmark, Claude/Codex provider and staged-write evidence, resumable repair state, hash-bound score/quality evidence, manifest, parity task/campaign, evidence, routine manifest/control/history/service-grant/service-state, and model-workflow receipt artifact types.
- Captures runtime state, logs, outputs, decisions, and evidence outside the repository by default under `~/.codex/conductor/workspaces/<workspace>-<hash>/`.
- Preflights workflows into resumable prepared runs with redacted `launch.json` manifests, and can launch an explicitly requested detached per-run worker only after that complete preflight succeeds.
- Records strict `background.json` lifecycle metadata and dedicated worker logs; approval values cross once through an anonymous pipe and are never persisted in worker argv, environment, launch metadata, or logs by the handoff layer.
- Optionally sends one content-free desktop notification when a direct, saved, model-authored, automatic, goal, detached, or scheduled-routine execution reaches `blocked`, `completed`, `failed`, or `stopped`. `--desktop-notify` is explicit, unavailable backends fail open, and strict per-status receipts suppress replay without retaining task text, output, paths, errors, or execution identity.
- Runs bounded adaptive foreground goal loops with `run-goal`, using a command verifier, a fail-closed read-only Codex verdict, or an ordered command-then-model chain; failed commands short-circuit the model call and bounded redacted feedback reaches the next iteration's agent prompts.
- Records run, phase, step, and `agent_map` packet trace timing, step kind, process output metrics, and bounded agent/item counts for local visibility.
- Holds one execute-wide run ownership lock and records per-run `runner.json` heartbeat/owner metadata with runner and active child process/session identities but no commands; dashboard/live/API/status summaries compute liveness and expose bounded active-child counts.
- Checkpoints each single `codex_exec` thread as soon as `thread.started` arrives. One stable thread plus exactly one locally observed `turn.completed` can additionally persist strict output-, checkpoint-, telemetry-, workspace-path-, and optional direct-write-workspace-bound terminal evidence before process return, allowing ordinary retry/resume to finish locally with no provider replay. If that final evidence never became durable, explicitly approved hard-loss recovery can still start a new continuation turn on the same hash-bound Codex thread without replaying the original prompt.
- Preserves useful failed-turn output instead of losing it. When ordinary `codex_exec` returns nonzero or times out, the runner normalizes any final-message file or, only from one strict terminal JSONL stream, recovers completed agent messages into the workflow's expected bounded redacted artifact. Provider, step, and run failure remain authoritative; no retry, continuation, verifier, model call, token reservation, or permission is added. State and public views expose only strict-stream validity, terminal class, budget-exhaustion marker, source class, byte count, and output SHA-256. Dashboard and console readers rehash the current workflow-bound artifact without following symlinks and mark drift as invalid.
- Journals content-free progress for every ordinary `codex_exec` call and launched `agent_map` packet from the JSONL stream already required for execution. Current v4 additionally distinguishes runtime-authorized native delegation from passive observation and, after a successful native call, reconciles aggregate parent-plus-child gross and weighted usage from bounded local Codex metadata. It retains only lifecycle, fixed counters, aggregate token integers, configuration, timestamps, attempt identity, and a session hash. It never retains prompts, messages, reasoning, commands, output, paths, diffs, tool payloads, or raw sessions, and observer/reconciliation failure never changes provider execution or recovery authority.
- Automatically records structured Codex CLI session events and measured token usage, and accepts operator-supplied or imported provider usage/cost evidence.
- Applies optional per-call and per-map runtime-hard weighted rollout budgets to ordinary `codex_exec` and `agent_map` steps. An explicitly authorized native step shares one aggregate parent-plus-child budget, caps one to six concurrently open children at depth one, and remains read-only. Generated maps group up to 1,000 validated source items under a 25-packet ceiling, cached packets consume no fresh allocation, and unsafe or underfunded fan-out fails before provider launch.
- Defines reusable `conductor.agent_profile.v1` contracts for `codex_exec`, `agent_map`, and `agent_team` members, with specialty instructions, locked model/effort, sandbox/risk/resource ceilings, bounded hash-bound workspace context, opt-in isolated or command-restricted tools, and bounded workspace-scoped cross-run memory. Memory defaults to newest-first selection or can opt into deterministic lexical relevance using the immutable profile/workflow task scope, rare-term weighting, tag boosts, and recency tie-breaking. Restricted profiles add structured exact/prefix Bash rules, optional `apply_patch`, operator-authorized selected Skill scripts, and operator-pinned HTTPS or sandboxed Skill-local stdio MCP dependencies behind one hash-bound session hook. Selected HTTPS routes may use separately approved bearer or HTTP-header credentials sourced only from named environment variables; authenticated and stdio routes support direct, map, team, and guarded same-thread continuation with fresh per-call capabilities. Native Codex memory remains disabled for isolated workers; Conductor injects an immutable redacted external snapshot as untrusted context. Steps may narrow profile authority but cannot widen it, and snapshot changes invalidate new-run packet caches without changing an already prepared run.
- Runs bounded `agent_team` steps with a configured lead, independent persistent member sessions, a dependency-ordered shared task board, deterministic parent-mediated claims, teammate mailboxes, per-turn and aggregate token ceilings, resumable checkpoints, fresh-reset generations, a synthesis report, and redacted CLI/dashboard inspection. Read-only remains the default. Teams may opt into active messaging: each direct or allowed broadcast message deterministically creates a bounded recipient response task, wakes that teammate in a later round, reuses its session, and can continue only within fixed task, hop, round, turn, and token limits; omission leaves passive messages unchanged and creates no response call or scheduler work. A separate `operator_chat` policy lets a teammate ask a bounded human question: the turn cannot commit unless a deterministic response task already fits every remaining capacity limit, other work continues, and a secret-screened CLI or live reply wakes the asking teammate in its existing session under a narrower cap. Timeout preserves the waiting task for explicit reply and retry; omission creates no chat path, lock, poll, prompt branch, or provider work. Optional `operator_console` atomically streams a strictly bounded private teammate view containing only lifecycle markers, tool-kind/status metadata, and redacted assistant messages; hash-only CLI/public dashboard/API summaries stay content-opaque, while explicit read/follow or a token-exchanged private auto-refresh page reveals the retained text. The browser read grant is short-lived, signed, `HttpOnly`, `SameSite=Strict`, exact-transcript scoped, and cannot authorize redirects; omission adds no console-specific callback, transcript writer, or artifact I/O beyond the baseline hash-only terminal recovery observer. A workspace-write team may opt exact static tasks into pre-write plan approval: the assignee plans in a disposable read-only copy, then either the fixed lead reviews autonomously or the human operator approves/rejects without a review-model call. Rejection returns bounded feedback to the same planner session; only an approved plan reaches later same-session isolated implementation. A team may separately opt into a strictly capped operator inbox so `queue-team-task` can add a named teammate follow-up or interrupt that teammate's active provider turn and deliver a redirect in the same session; disabled teams create no inbox, lock, or interrupt poll. Interrupted output and isolated workspace changes are discarded, while the full authorized turn cap remains conservatively charged. Optional `team_task_created`, `team_task_completed`, and `team_member_idle` command gates receive a strict hash-only envelope before candidate state or source changes commit. A gate may opt into one to four same-session quality retries for selected exits; each retry uses the same execution workspace, including the retained isolated member stage for write teams, consumes the existing aggregate team budget, and is checkpointed without raw feedback or session IDs. An opt-in `workspace-write` team gives each active member an external isolated copy, then the parent alone merges changes in stable team order; identical overlaps are deduplicated and different same-path edits rerun only the later task serially against the merged source. A lead may add a strictly capped ordered set of tasks without changing members, tools, permissions, rounds, workers, or budgets; a failed round preserves successful sibling work for explicit retry.
- Displays all or selected private teammate turns in a dependency-free native terminal surface with `team-console`: responsive stacked/multi-column layouts, bounded event/message/history tails, strict JSON, and deduplicated follow mode. Terminal control characters are rendered inert; the command is read-only and uses no model call, network, background process, or additional transcript writer.
- Imports redacted local JSON/JSONL session trace summaries, including common local provider export shapes, into run state for dashboard/live agent-session visibility.
- Writes and validates durable routine manifests, controls, and bounded history; supports manual, interval, daily, weekdays, weekly, monthly, one-shot, and strict five-field numeric cron schedules, one-most-recent catch-up, run-now, pause/resume, schedule edits, foreground/detached scheduling, and an optional explicitly installed restart-persistent user service.
- Wraps Codex Cloud submit/status/review/apply operations behind explicit network/write approvals; apply requires a matching review receipt and an unchanged diff hash.
- Applies bounded no-follow reads to workflow metadata, run state, prompt files, templates, cache metadata, dashboard excerpts, and cache fingerprint hashing so malformed or oversized local files fail safely.
- Runs optional policy-gated local workflow hooks around run, step, team-task, and every Codex provider process launched inside a declarative workflow. Provider hooks can target steps, profiles, invocation scopes, members, and tasks through strict content-free metadata. Opt-in `agent_pre_tool` hooks additionally gate Codex-supported native tool calls through trusted session `PreToolUse`; omission preserves the original provider command and adds no hook process or model call.
- Generates validated workflow drafts and safe saved workflow commands from task text, failing commands, verification commands, and packet item sources, with packetized drafts capped at 1,000 source items, 25 model packets by default, and 128 workers; actual concurrency above 16 requires `--approve high-scale-agent-map`.
- Uses `plan-model-workflow` to let read-only Codex inspect a workspace and author a task-specific declarative workflow draft behind `--allow-agent`; explicit review planning binds a bounded before/after workspace content snapshot by default, then writes the draft and provenance receipt outside the repository.
- Uses `run-reviewed-model-workflow` to preflight and execute that exact current receipt-bound graph with zero replanning; graph, receipt, wrong-workspace, or tracked-content drift fails before a run receipt or worker exists.
- Uses `run-model-workflow` for an explicit task-to-plan-to-preflight-to-run handoff in foreground or `--background` mode; it requires `--approve model-workflow-execute`, preserves every generated manual gate and capability check, and writes a separate validated run receipt.
- Uses `run-model-goal` for bounded adaptive work in the foreground or one explicitly requested `--background` worker: each unsatisfied verifier result causes a fresh model-authored graph while the planner continues in the same bound Codex session by default; opt-in `--dynamic-interval` makes that same model-only verifier select a persisted 1-60 minute retry deadline without another provider call, and optional `--monitor-command-json` can wake early on the first non-empty line from one policy-checked read-only process; resume binds the original task, planner configuration, verifier, delay/monitor policy, and session mode fingerprints.
- Uses `auto` as a deterministic one-command front door for substantial work. Its serial default compiles one direct bounded Codex step with zero planning calls. Tightly bounded verifier-backed single-function repairs and named-file test additions can use `low`; other bounded verified work uses `medium`, while unverified, broad, sensitive, or complex work stays `high`, and explicit effort through `ultra` is preserved. With a verifier and explicit multi-worker envelope, it tries that direct attempt first and escalates to same-session model planning only after failure. `--strategy direct|progressive|workflow|goal` pins those policies; explicit `--strategy native` instead compiles one read-only parent with bounded depth-1 Codex-native children, no planner, and one shared aggregate token cap. Missing execution approval safely stops after preflight, and task text never controls routing or capabilities.

It accepts the existing JSON-compatible Conductor export or statically compiles common Claude `agent()`, `pipeline()`, `phase()`, and static `parallel(SOURCE.map(item => () => agent(...)))` shapes into the audited runtime without evaluating JavaScript. Bounded `+` text composition, `whenToUse`, and static per-agent effort are preserved; Claude `max` effort maps to Codex `ultra`. Static parallel callbacks become true bounded `agent_map` fan-out, and pre-rendered prompts use hash-bound opaque items instead of filesystem paths.

## What It Does Not Do

- No executable code inside the text-only Skill folder; the companion runtime ships beside it in the release bundle.
- No runtime network dependencies.
- No arbitrary JavaScript workflow execution. Saved `.claude/workflows/*.js`/`.mjs` files are either parsed for JSON-compatible Conductor workflow data or accepted by a strict non-evaluating compiler limited to static metadata/constants, bounded top-level `agent()`/`pipeline()`/`parallel()`/`phase()` forms, structured arguments, exact map-result handoffs, and `return`.
- No auto-updates.
- No hidden agents.
- No hidden, implicit, or system-wide workflow daemon and no arbitrary process-manager interface. Background run, model-goal, and detached routine-supervisor workers remain visible one-shot launches with no restart policy. The optional routine service is installed only by its dedicated approval-gated command as one visible launchd LaunchAgent or systemd user unit over exact expiring grants.
- No managed background model-goal scheduler, daemon, or restart service. `run-model-goal --background` launches exactly one visible bounded worker after explicit invocation and full preflight; model verification remains read-only and fail-closed.
- No implicit or unapproved workflow execution. `plan-model-workflow` remains plan-only; `run-model-workflow`, `run-model-goal`, and `auto` execute only with `model-workflow-execute` and after complete policy preflight. Without that approval, serial `auto` deterministically compiles and preflights with zero provider calls.
- No provider-native text injection into a still-running teammate turn, provider-native same-turn peer delivery, takeover of the provider's native interactive teammate session, or exact recovery of a provider turn that was in flight during hard process loss. Opt-in operator console provides a separate private redacted transcript tail; active messaging provides deterministic restart-persistent next-round peer wakeups; operator chat provides bounded between-turn questions and same-session replies; current-turn interruption uses selected-child stop plus same-session redirect. Plan approval can use autonomous fixed-lead review or an explicit no-review-call operator decision. Automatic quality retry is explicit workflow policy and bounded by hook and team budgets. Team no-replay recovery begins when a locally observed terminal event and exact output/workspace checkpoints are durably bound, before provider-process return and quality acceptance; an in-flight result or a remote completion lost before that local receipt remains uncertain.
- No deployment, commits, pushes, production writes, or destructive actions without chat-visible approval.

## Clean-Room Statement

This project is implemented from `PROJECT_SPEC.md` only. It does not copy third-party skill code, prompts, assets, or runtime design.

## Install

Extract the release bundle, validate the no-write plan, then install:

```bash
unzip codex-conductor-bundle.zip -d codex-conductor-release
python3 codex-conductor-release/install.py --dry-run
python3 codex-conductor-release/install.py --allow-writes --approve conductor-install
```

The installer verifies `release-manifest.json`, writes versioned and active copies under `~/.codex`, records a local receipt, and is idempotent. It does not modify a repository, shell profile, PATH, service, or scheduler. Replacing differing installed bytes additionally requires `--replace --approve conductor-update`.

For native Codex plugin discovery instead, extract and add the local marketplace:

```bash
unzip codex-conductor-marketplace.zip -d codex-conductor-plugin
codex plugin marketplace add codex-conductor-plugin/codex-conductor-marketplace
codex plugin add codex-conductor@codex-conductor-local
```

The plugin embeds `conductor-runtime.pyz` under its `scripts/` directory; it installs no dependency or service and uses the same external Conductor state root. The standalone Skill/bundle and plugin are alternative distribution paths, so install only one Skill source at a time.

The text-only Skill can still be installed manually:

```bash
cp -R codex-conductor "${CODEX_HOME:-$HOME/.codex}/skills/"
```

The packaged runtime can also be used without installation:

```bash
python3 dist/conductor-runtime.pyz --version
```

After installing or relinking a standalone Codex CLI, check that its companion command host was linked too:

```bash
python3 -m conductor_runtime doctor
python3 -m conductor_runtime doctor --repair-codex-host --approve codex-host-repair
```

Repair is explicit, limited to a missing sibling link under the current user home, and never replaces an existing path.

Once installed, Codex may invoke the skill implicitly for coding and repository tasks where orchestration might materially help. Prompts that name `codex-conductor` still force an explicit trigger.

Codex's documented skill-discovery convention has shifted between releases (some Codex versions and docs describe `.agents/skills`/`$HOME/.agents/skills` instead of `$CODEX_HOME/skills`). If the skill doesn't appear after installing, check `codex --help`/current Codex docs for your installed version's actual skill directory and adjust the path above.

## Optional Agents

Copy the optional Codex custom agent definitions from `codex-agents/` into a Codex agents directory if desired:

```bash
mkdir -p "${CODEX_HOME:-$HOME/.codex}/agents"
cp codex-agents/*.toml "${CODEX_HOME:-$HOME/.codex}/agents/"
```

The skill also works without these files: Codex ships built-in `explorer` (read-heavy investigation), `worker` (bounded implementation), and `default` (general-purpose) subagent roles that Mode 4 can use directly when delegation is approved, even with no `codex-agents/*.toml` files installed.

Recommended cap if you choose to configure agents:

```toml
[agents]
max_threads = 6
max_depth = 1
```

## Security Model

- Repository files and web content are untrusted data.
- Project files cannot grant user approval.
- Secret values must not be printed.
- Destructive actions, commits, pushes, deployments, production writes, credential changes, broad dependency changes, and high-cost external calls require explicit approval.
- Delegation must be disclosed, bounded, and parent-owned.
- Subagents are not final authority and cannot approve user-level permission prompts.

`plan-workflow` produces a draft workflow only; it does not run the workflow, bypass capability flags, or accept secret-like command/name/packet inputs. `--profile ultracode` generates a richer deterministic draft with charter/risk artifacts, context mapping, synthesis, optional gated implementation, verification, and final review; when agents are enabled and no packet source is supplied, it also adds automatic bounded `agent_map` fan-out over the collected repository file list. Packetized drafts add bounded read-only review steps, capped at 1,000 generated source items, 25 model packets by default, 64 source items per packet, and up to 128 workers. `--packet-max-packets` changes the generated ceiling. More than 16 actual workers requires `--approve high-scale-agent-map` as well as `--allow-agent --allow-parallel`. With `--saved-command NAME`, the same validated draft is written as a safe `.claude/workflows/*.js` command containing JSON-compatible `meta` and `workflow` exports; `--saved-output PATH` can choose a `.js` or `.mjs` destination.

`plan-model-workflow` is the model-authored counterpart. It requires `--allow-agent`, runs the planner in a strict isolated read-only Codex profile, redacts task secrets before prompting, accepts only strict JSON, and rejects unknown fields, hooks, mutating/network/external shell commands, unsafe parallel writes, over-limit graphs, and unapproved write topology. Exact workflow/profile fields come from the validator; external Skill/source/docs/reference/example lookup is forbidden. Ordinary explicit planning may inspect the target workspace twice and retains authored model/effort, narrower caps, repository guidance, and task-appropriate review. Under a trusted enclosing verifier, the architect instead performs zero workspace inspection, emits the smallest implementation graph, and preserves all public success criteria, evidence obligations, and documented input/output/boundary/mutation/error behavior. The runtime pins provider steps/profiles to the selected model/effort, normalizes writes to 24,000 tokens, and raises undersized provider captures to 256 KiB. V12 additionally appends a final trusted worker contract after authored prompts: the call is the complete workflow; external skills/plugins/goals/orchestration are off-scope; tests, runtimes, toolchain probes, diff inspection, and bespoke checks belong to the enclosing verifier; the first write must not be deferred past the half-budget checkpoint; and tool use stops after complete writes plus static contract audit. This directly prevents nested Conductor invocation and duplicate verification without raising caps. Architect caps remain 4,000/8,000/12,000/16,000/20,000 for `low` through `ultra`. Generated writes remain approval-gated and bound receipts retain only hashes/counts. Current planning uses `conductor.model_workflow_prompt.v12`; frozen v11 and earlier accepted contracts remain readable, serial direct uses prompt v4, and native direct remains v3.

`run-reviewed-model-workflow WORKFLOW PLANNING_RECEIPT --workspace PATH` executes the inspected graph without invoking the planner again. For a bound v4 receipt it compares the canonical workflow name, filename, fingerprint, mode, and step count, derives every planning limit and write-capable bit from the receipt, verifies the resolved workspace path and fresh bounded content snapshot, and only then enters the same full preflight plus foreground or detached execution path as `run-model-workflow`. It accepts no task, model, effort, topology, or limit overrides. `--allow-agent` is required only when the reviewed graph contains `codex_exec`, `agent_map`, or `agent_team`; all other capability and manual gates remain graph-derived. V3 and unbound-v4 compatibility preflight requires `--allow-unbound-workspace`; actual execution additionally requires `--approve unbound-reviewed-workspace`. The run receipt binds the exact planning-receipt bytes used for those decisions.

Every ordinary `codex_exec`, `agent_map`, or `agent_team` turn with a configured weighted-token ceiling receives a completion-reserve contract. The runtime's 50%-remaining reminder is a checkpoint: completed work finalizes with concise evidence, while only a concrete unmet criterion justifies using the reserve. Map/team aggregate allocation may lower a call below its configured per-call ceiling, so prompts state that ceiling as an upper bound and leave the actual reminder authoritative. Interrupted single-agent continuation receives the same rule instead of reopening optional exploration. Deterministic serial read-only direct prompt v4 now consolidates multiple methods or call sites in one component that bypass the same required helper into one component-level root, citing the earliest faulty call and retaining the others as evidence. Native direct turns use the non-duplicated `parent-integration-and-synthesis-v4` contract instead: the parent must inspect a complementary file group while each child handles only one independent group, stop or interrupt unfinished child work at 50% remaining, and enter no-tools final synthesis at one-third remaining capped at 6,000 tokens. Diagnosis/review synthesis consolidates parent/child overlap by violated invariant or independently actionable repair. Separately repaired components stay separate. Candidate findings sharing one first faulty executable line or repair site are merged; one missing validation boundary stays one finding across its field/type/value/normalization/exception manifestations, which move into evidence. The requested output contract is checked once before returning. This schedules model behavior inside the existing hard cap; it does not earmark provider tokens per session.

`auto` is the high-level front door over the existing workflow and adaptive-goal primitives. Its `deterministic-v5` route is independent of task text. The default one-worker envelope compiles a strict direct workflow without a planning provider call; a verifier selects direct adaptive retries. Native mode is explicit and read-only. It defaults to one concurrently open child; `--native-agent-threads 1..6` changes that child-only limit while the parent remains separate. It replaces the generic completion prose for native turns instead of duplicating it. Both checkpoints are operator-visible but are not provider-enforced per-session reserves. Irrelevant plugin, app, browser, MCP, web, hook, goal, memory, and dependency surfaces are disabled while repository guidance and the selected model remain intact. It requires `--allow-agent --allow-parallel --approve native-agent-delegation`; execution additionally requires `--approve model-workflow-execute`. Plan-only compilation makes zero provider calls. Ordinary workflow agents can be bounded without using a comparison adapter. Top-level `agent_effort`, `agent_max_tokens`, and `agent_map_max_total_tokens` set workflow defaults; `codex_exec` accepts `effort` and `max_tokens`, while `agent_map` accepts `effort`, `max_tokens`, `max_total_tokens`, and `max_packets`. Token values are runtime-hard weighted rollout limits, not dollar budgets or gross terminal-token limits. `max_packets` deterministically groups contiguous validated source items into JSON-array scopes and permits at most 64 items in one call. The map total is divided fairly over only uncached packets and combined with the per-call cap. An impossible packet ceiling or underfunded share fails before provider launch. Each grouped cache key binds every member's bounded content fingerprint, and `retry-packet` targets the complete grouped scope. An explicit retry is a new bounded attempt, while a hard local interruption can still leave prior remote spend uncertain. Deterministic and model-authored planners use the `balanced-v2` profile with at most 25 map packets; all fields remain optional for existing hand-authored workflows.

`codex_exec.context_from` transfers actual outputs from explicit direct dependencies instead of treating `depends_on` as an evidence channel. Supported sources are `write_artifact`, captured `shell`, `codex_exec`, and `agent_map`. The runtime verifies producer receipts or complete map-cache bindings, source freshness, output hashes, regular no-follow files, and fixed size/count limits before launch. It then injects one redacted, marker-escaped, path-free untrusted-evidence section capped at 64 KiB; state records only counts, bytes, and a hash. Generated Ultracode workflows use this for context, analysis, synthesis, implementation, verification, and final-review handoffs; final review receives the synthesis contract before implementation and verification evidence. Omitting `context_from` preserves the prior lightweight path and adds no receipt hashing or prompt content.

`codex_exec.completion_verdict: strict-v1` turns a read-only review call into a fail-closed acceptance gate without adding a provider call. The model must return one strict bounded JSON verdict; malformed or unsatisfied output fails the step and blocks downstream work even when provider execution succeeded. Hashing and parsing use one no-follow file snapshot. State, run detail, and diagnostics expose only status, counts, confidence, bytes, and hashes, while generic artifact and provider-log excerpts omit verdict text. Generated Ultracode final review uses this contract automatically. Inside an explicitly bounded `run-goal` or `run-model-goal`, one valid false verdict may feed its private redacted feedback into the next authorized iteration; the failed run stays failed, the external verifier is skipped for that iteration, malformed or drifted evidence stops fail-closed, and success adds no repair work. Ordinary steps that omit the contract retain prior behavior.

`run-goal --workflow-completion` uses the workflow's existing strict verdict as its verifier and launches no external command or model review. It is explicit and mutually exclusive with every external verifier flag. Preflight requires exactly one `strict-v1` step and requires it to transitively depend on every other workflow step, so an early or parallel review cannot authorize completion. A completed run is accepted only after the runtime independently reopens the private artifact and checks workflow fingerprints, step state, exact bytes/hash, strict parsing, satisfaction, confidence, and evidence count. Goal state and public detail retain only zero-call status and hash/count metadata. Generated Ultracode workflows satisfy this topology automatically.

`run-model-workflow` is the explicit automatic plan-and-run handoff. Before spending on planning it checks agent, write, parallel, high-scale, and fixed execution approvals. After planning it executes the entire graph once in dry-run mode so shell, agent, risk, and manual-gate policy failures stop before workflow processes start. It then either resumes the same run in the foreground or, with `--background`, records a prepared launch and starts one isolated worker. `--dry-run` stops after preflight and does not require `model-workflow-execute`. If planning times out or fails, no graph is preflighted or executed and a strict failed-attempt receipt records task hash, effort, limits, and error class. The `conductor.model_workflow_run_receipt.v3` contract supports foreground and detached execution; a detached receipt starts in `running`, is hash-bound in the anonymous launch payload, and is atomically finalized by the worker. Its path and approval values are not persisted in background metadata. Strict `v1` and `v2` receipts remain validatable.

`run-model-goal` extends that handoff across bounded iterations. Each failed command verifier, unsatisfied read-only model verifier, or valid unsatisfied in-workflow completion verdict produces bounded feedback, and the next iteration asks the read-only planner for a new graph against the current workspace. It binds to loopback by default, prints a per-server control token in the terminal, auto-refreshes run, goal, and benchmark summaries, links each goal to a selected-goal detail page through an opaque handle, links each run to a selected-run detail page, provides `/api/run/<run-handle>` for bounded redacted selected-run detail JSON with artifact/log excerpt text omitted by default, provides a redacted workflow JSON download for the selected run, shows prepared-run launch handoffs, manual-gate approval guidance, advisory runtime action summaries, runner heartbeat liveness (`alive`, `stale`, `finished`, or unavailable), active child-process counts, diagnostics, run/step durations, separate agent item/packet/call counts, and bounded `agent_map` trace evidence when recorded, and delegates guarded interrupted-run recovery, packet retry, step retry/reset/skip, plus run-level pause/resume/stop/restart/terminate actions to the same audited controls exposed by the CLI. Selected-run packet rows can issue token-gated canonical-scope-hash-bound retries for failed, timed-out, output-missing, cached, or completed packets while preserving compatible sibling cache entries. Runtime action summaries classify ready, dependency-waiting, operator-action, failed-recovery, and active steps without executing anything. The token is not embedded in the page and does not grant manual-gate approval; paste it into a control form only when taking a supported run/step control action. Binding to a non-loopback host requires `--allow-remote` and an external security boundary; the runtime does not provide production TLS or multi-user authentication. Runtime-launched Codex steps report measured token totals when the CLI emits them; other token/cost totals remain unavailable until recorded or imported from a measured source.

Private transcript cookies intentionally omit `Secure` because the default server is loopback HTTP. Any explicitly remote deployment must terminate TLS at its external boundary and prevent direct unencrypted access to the runtime port.

`skip-step` does not bypass `manual_gate` approvals. Every workflow execution holds an exclusive no-follow execution lock, so a second resume and direct retry/reset/skip controls fail before racing the active owner. Run-level pause, stop, and restart are honored at step boundaries; they do not kill or restart an already-running subprocess. `terminate-run` is the explicit OS signal path: it refuses non-active or terminal runs, missing/inactive/tampered runner metadata, mismatched run status, heartbeat metadata older than 300 seconds or materially future-dated, other-host metadata, the current process/group/session, and non-session-isolated foreground runners before signaling the recorded process group. After hard interruption, `recover-run` requires POSIX cross-process lock proof, same-host runner metadata, an inactive owner, and no live recorded child session. With no resolution flag it records the uncertain step failed; `--retry-running` resets only that step for a fresh invocation. For exactly one interrupted `codex_exec` with an active matching checkpoint, mutually exclusive `--resume-codex` binds the approved checkpoint bytes and lets the next normal `run --resume` start a new turn on the same Codex thread. It does not resume the exact in-flight turn or prove whether that remote turn completed. A crash between steps is reconciled directly to resume. Handled exceptions finalize failed state automatically and do not require recovery. `restart-run` resets every workflow step to `pending`, preserves existing artifacts/logs, and invalidates `agent_map` cache generations and Codex continuation bindings. `save-run-workflow` exports the run's validated, redacted workflow JSON to a new destination and refuses overwrites or symlink destinations. Runtime-launched Codex agents use `--json`; normalized event metadata and terminal measured usage are recorded automatically without raw prompts, message text, or tool arguments. `import-usage` and `import-session-trace` remain available for bounded local provider artifacts and optional measured cost data; they do not call billing APIs or estimate cost.

Read-only and direct-write paths perform no prelaunch workspace scan; direct-write evidence takes one terminal snapshot and one recovery snapshot. Generic artifact views suppress the private body, while validation, inspection, schema, static/live/API, and diagnostics expose hashes and bounded totals only. Missing or unstable terminal evidence preserves the existing post-return or explicit same-thread-continuation behavior.

The self-hashed artifact binds exact workflow/step/generation/round/member/task/retry identity, launch state, retained output, raw resumable session, bounded provider telemetry, write-workspace base/result evidence, session mode, base/effective prompt hashes, and any lifecycle-context receipt; retained v1 remains readable. It binds the exact prior and candidate team state, prior ledger, expected source, retained output, isolated workspace, launch-base manifest, and hashes. Partial conflict state/ledger checkpoints, accepted quality retries, and reset-before-new-generation are recovered deterministically. `validate-agent-team-merge-intent` and `inspect-agent-team-merge-intent` expose only bounded identity and hashes.

Before an accepted merge can mutate source, the parent atomically writes a strict pending transaction that binds the exact merge plan, source-before manifest, retained teammate workspace, and before/candidate state and ledger hashes. Resume accepts only exact before, partial, or candidate source checkpoints and exact before/candidate metadata; it deterministically completes the remaining source writes, commits candidate state and ledger, emits a no-provider-replay recovery receipt, and removes the completion, intent, and transaction markers. Terminal/completion/intent/transaction/recovery validation and inspection, public schemas, reports, metrics, and dashboards expose bounded evidence without file contents. ## Limitations

- The installable Skill folder is text-only; the release bundle ships the companion runtime beside it as `conductor-runtime.pyz`.
- The runtime calls Codex only with explicit agent flags; isolated comparison adapters additionally require network and `codex-live-run`, and the staged path requires write permission plus `codex-staged-write`. Claude calls use the paid read-only or staged adapters with agent, network, Sonnet/Ultracode, `claude-paid-run`, and any required write gate.
- It may be implicitly selected as a skill and can recommend the `auto` front door, but implicit selection alone does not grant agent, execution, write, network, or paid-provider capability and never installs dependencies or contacts external services.
- Validation tools are development-only and are not included in the installed skill package.
- The project is designed to compete with Ultracode-style orchestration. The homogeneous v0.82 implementation campaign supports the preregistered product-profile broad-superiority gate, but strict equivalence and universal superiority remain unproven because cap mechanisms differ, one row is outcome-incomparable, Codex dollar cost is unavailable, and only five repeated task families are covered. The parity matrix smoke suite validates interfaces and evidence fixtures only.
- Provider lifecycle hooks observe local process boundaries, not individual tools or an already-running provider turn. `agent_start` can block before launch and inject bounded receipt-backed context into fresh/resumed direct calls, map packets, and team plan/review/turn/quality-retry calls. Explicit blocking `agent_stop` hooks are crash-safe acceptance gates for locally durable results. Matching direct writes are transactionally isolated until acceptance. Separate `agent_pre_tool` hooks gate only Codex-supported `PreToolUse` paths; upstream incomplete coverage prevents a universal interception claim, and no transaction can recover a provider result that lacks exact local terminal evidence.
- Ordinary `codex_exec` steps and cacheable `agent_map` packets can recover from exact locally durable final-turn evidence without provider replay. If that evidence is absent, one interrupted `codex_exec` may explicitly open a new turn in the same Codex thread. Exact in-flight continuation, remote completion lost before durable local evidence, and uncertain remote token accounting remain open.
- Native usage reconciliation depends on Codex's local thread index and rollout token-event format. It is bounded, read-only, content-free in retained evidence, and fail-open; when that private provider format changes or evidence is incomplete, views report `unavailable` and the shared runtime-hard weighted cap remains authoritative.
- Desktop notification delivery is opt-in across trustworthy local workflow, goal, wrapper, detached-worker, and scheduled-routine/service terminal paths. Cloud-task notification delivery remains open because completion may occur asynchronously off-host; live login/reboot delivery reliability is not yet established.
- Packet batching has deterministic scheduler, cache, retry, and 1,000-item synthetic coverage, but its quality/latency tradeoff has not yet been measured across a live task-diverse provider suite.
- Coordinated teams may be operator- or model-authored and remain read-only by default. An opt-in private console atomically follows redacted teammate lifecycle, tool-kind/status, and assistant-message events through a native multi-member terminal display, single-turn CLI tail, or token-gated auto-refreshing local page; named-member follow-ups are delivered at parent-controlled checkpoints, current-turn redirection stops one selected child and resumes its session, and bounded teammate questions wake the asking session after a secret-screened human reply. Opt-in write teams may use autonomous fixed-lead review or cheaper explicit operator approve/reject revision, then use external member copies, parent-owned deterministic merges, durable local-terminal recovery before provider-process return, local gates, and planning, and exactly-once source/state/ledger transactions. Provider-owned split-pane session takeover, recovery of a provider turn still in flight or remotely completed before its validated local terminal receipt, exact in-flight continuation, and matched live quality evidence remain open.
