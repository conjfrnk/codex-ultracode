# Expected Behavior

Codex Conductor should choose the least powerful workflow mode that safely satisfies the request, keep the parent session responsible for all integration and final claims, and avoid unsupported certainty.

## Normal Workflows

- An extracted release bundle validates a strict manifest before installation. Dry-run performs no writes; first install requires `conductor-install`; differing replacement additionally requires `--replace` and `conductor-update`; identical reruns are no-write successes. Runtime and Skill copies remain under configured external homes, symlinked/unmanifested Skill content fails closed, and installation never edits repositories, PATH, shell startup, services, or schedulers.
- The native local marketplace archive contains one available `codex-conductor` plugin whose version matches the runtime, whose Skill tree matches the standalone text-only Skill, and whose embedded script runtime matches the packaged `.pyz`. It declares no MCP server, app, hook, daemon, or credential dependency. Marketplace add/install/list and cached-runtime execution are tested only in an isolated Codex home; packaging never mutates the user's actual plugin configuration.
- Goal-driven coding or repository work should invoke Codex Conductor when skill metadata is available.
- Coding workflows where orchestration might materially help may use implicit invocation.
- Simple implementation work uses Mode 2 with no agents.
- Broad or resumable work uses Mode 3 and writes visible workflow artifacts.
- Agent workflows use Mode 4 only after a disclosed delegation plan.
- Bulk row workflows use Mode 5 only with explicit approval, schema, and concurrency limits.
- Codex Cloud task workflow uses Mode 6 only when available and useful (`codex cloud exec` to submit, `codex apply` to review and merge), not a live session handoff.
- Ordinary `codex_exec` and `agent_map` workflows may set validated effort and runtime-hard weighted rollout caps at workflow or step scope; absent fields preserve legacy command behavior.
- A `codex_exec` or `agent_map` may set one bounded JSON-compatible `output_schema`. The runner materializes it as a private no-follow run artifact before provider launch, passes only that path through Codex `--output-schema`, and requires an existing resume artifact to remain byte-identical. An `agent_map` may pair `items_artifact` with a bounded strict `items_pointer`; duplicate/nonstandard JSON, invalid pointer escapes or traversal, missing values, non-string arrays, symlinks, oversized files, and ordinary packet-item policy violations fail before any map provider starts. Omission preserves the old provider command and line-oriented item artifact behavior.
- Saved `.claude/workflows/*.js`/`.mjs` commands accept either the existing JSON-compatible `meta`/`workflow` exports or a non-evaluating static Claude-style subset. The latter permits static `meta`, top-level awaited `agent()`/`pipeline()` bindings, literal or structured-arg item arrays, prior-agent required array-of-strings schema handoff, callback template interpolation, static schema/label options, and direct or pipeline-filter return. It compiles every step read-only, preserves top-level order, schedules one packet per item, limits concurrency to 16 and total possible agents to 1,000, and uses structured Codex output plus strict JSON-pointer handoff. Imports, loops, branches, mutation, arbitrary calls, computed expressions/interpolation, unsupported options, forward references, missing schemas, and trailing script bodies fail closed without Node or JavaScript evaluation. Normal agent/parallel approvals remain required.
- A `codex_exec` may opt into `context_from` for one through 32 unique direct dependencies that produce artifacts (`write_artifact`, captured `shell`, `codex_exec`, or `agent_map`). Referenced single-output producers write hash/size receipts only when needed; maps reuse their complete cache/source/output binding, and dynamic maps must declare a packet ceiling whose combined upper bound fits 32 artifacts before any process starts. Before provider launch, every source must be completed rather than skipped, every output must remain regular, no-follow, receipt-bound, and at most 2 MiB, and map source drift must remain cache-compatible. The exact dependency section is redacted, marker-escaped, path-free, capped at 32 artifacts, 8 KiB per excerpt, and 64 KiB total, then enters the ordinary prompt/checkpoint hash; state and run detail retain only contract, source/artifact counts, byte count, and SHA-256. Missing, stale, oversized, symlinked, drifted, or modified evidence fails before a downstream provider starts. Omission adds no receipt hash, prompt content, or provider call.
- A read-only `codex_exec` may opt into `completion_verdict: strict-v1`. The fixed final prompt requires exactly `satisfied`, `feedback`, `evidence`, and `confidence`; duplicate keys, non-finite values, malformed shape, over-limit text/lists, satisfied-without-evidence, and unsatisfied-without-feedback are invalid. Provider success remains separate: only a valid satisfied verdict completes the step, while invalid or unsatisfied output fails the step/run and blocks downstream work without another call. Hashing and parsing share one regular no-follow snapshot. The redacted review artifact remains private from generic artifact/log excerpts; state and run detail retain only validity, satisfaction, confidence, evidence count, feedback presence, bytes, and SHA-256, and diagnostics report rejection or drift. Local terminal recovery applies the same gate with zero provider replay. Generated Ultracode final review always uses this contract; omission preserves ordinary `codex_exec` behavior.
- An explicitly bounded `run-goal` or `run-model-goal` may treat exactly one valid unsatisfied in-workflow completion verdict as retryable private feedback. The failed run remains failed, no external verifier runs for that iteration, and the next already-authorized iteration receives only bounded redacted feedback framed as untrusted. Goal state and public detail retain the step/contract/confidence/count/size/hash receipt but no feedback or evidence text. Resume revalidates the copied workflow, failed state, one-read verdict snapshot, receipt, and goal binding before recovering feedback. Invalid, satisfied, missing, ambiguous, changed, or over-limit evidence never enters this repair path. Omission and first-pass success add no scan, prompt content, provider call, or retry.
- `run-model-goal --dynamic-interval` requires a model-only verifier and binds operator-selected minimum/maximum delays within 1-60 minutes into the goal fingerprint and detached handoff. The same strict completion-verifier call must choose one in-range integer delay and bounded reason only when unsatisfied, adding no provider call. The runtime persists the absolute UTC wake time before sleeping; interruption and matching resume reuse that time without reselection. Optional `--monitor-command-json` binds one secret-screened, no-shell, policy-classified read-only argv and treats the selected wake as a maximum deadline. Its isolated process wakes on the first bounded non-empty stdout line, then is killed; the redacted event enters the next planner turn as untrusted feedback while public views retain only hashes/counts. Timeout falls through at the deadline, nonzero or eventless exit fails closed, and interruption leaves an uncertain receipt that resume never replays. Verdict, reason, timestamp, bound, monitor, or payload drift fails closed before another iteration. Omission preserves immediate retries, ordinary dynamic sleeping, and the legacy four-field verdict exactly.
- `run-goal --workflow-completion` may replace the external verifier only when validation finds exactly one `strict-v1` step and its transitive ancestors are every other workflow step. The flag is mutually exclusive with command/model verifier options. A completed workflow is checked locally by reopening the bound private output and revalidating workflow fingerprints, satisfied state metrics, exact bytes/hash, strict parse, confidence, and evidence count. The goal check records `provider_calls: 0` and hash/count metadata without feedback or evidence text. Invalid/drifted evidence fails closed; an unsatisfied verdict uses the existing private repair path. Omission preserves the external-verifier requirement.
- A successful explicitly native `codex_exec` reconciles bounded content-free parent-plus-child gross and weighted usage from local Codex metadata; schema/path/topology/token failure reports unavailable without changing execution, and non-native calls do not inspect that metadata.
- Isolated Codex comparison adapters keep CLI-default service routing unless `priority` is explicitly selected. Current evidence binds the selected tier and binding method; priority adds an increased-usage notice and never changes effort, caps, permissions, retries, or claim gates.
- Strict reusable agent profiles may bind specialty instructions, model/effort, weighted-token/worker/timeout ceilings, sandbox ceiling, risk floor, bounded workspace context, and explicitly selected progressive Skill trees to `codex_exec`, `agent_map`, or `agent_team` members. Steps may narrow but never widen or replace the bound profile contract.
- Restricted profiles may activate an unauthenticated HTTPS MCP dependency only when an already-selected Skill declares the exact name, transport, and URL parsed by Codex; only operator-listed tools are exposed, each run requires network plus dependency-specific approval, configuration is process-local, and profiles without routes keep policy v1 without a discovery process.
- An authenticated selected-Skill MCP route additionally requires credential-specific approval and resolves only bounded environment-backed bearer/header values in the parent. Every actual direct, map, team, or same-thread provider call receives only a random opaque loopback route; selected values, their names, and inherited aliases are absent from Codex. The in-process broker pins the approved HTTPS target, strips client auth, denies redirects and upstream auth challenges, bounds transport, exact-redacts allowed headers plus buffered/SSE bodies, closes stalled sockets, and retains no daemon, config, OAuth store, transcript, or repository state. Arbitrary transformed, encoded, or split endpoint output remains outside exact-redaction guarantees.
- A selected-Skill stdio MCP route requires the general dependency approval plus `skill-mcp-stdio:<name>`, matches one expansion-free parsed command to an executable bound `scripts/` resource and exact args, and grants no network permission. Each direct, map, team, or same-thread provider call launches only a receipt-bound Conductor proxy over a fresh immutable external Skill snapshot. The child receives a sanitized environment and private temp under macOS Seatbelt or Linux bubblewrap; repository/home access, network, sibling execute bits, parent environment credentials, persistent config, and repository-local state are denied. Snapshot/receipt tampering and unsupported hosts fail closed.
- A profile may select at most eight canonical workspace-relative Skill directories. Each regular, symlink-free, non-empty UTF-8 `SKILL.md` entrypoint is loaded with a 128 KiB cap; up to 256 optional regular resources per Skill are recursively path/size/mode/hash-bound under depth, per-file, aggregate-file, and aggregate-byte limits. Bindings persist no raw Skill or resource content.
- Selected Skill resources remain out of the prompt until read on demand through existing workspace tools. Add/delete/content/mode drift fails before or after every provider call and invalidates resume, cache, routine, and authorization identity. Tree binding alone does not execute scripts or grant integrations, network, tools, or permissions.
- A restricted profile may separately authorize up to 32 non-empty executable resources under `scripts/` from its already-selected Skills, with exact arguments or an explicit argument prefix. Derived routes share the existing 64-rule command policy, never auto-run, and grant no other resource; direct, map, team, resume, cache, routine, trace, and dashboard evidence bind their count and exact policy hash.
- Direct, map, team plan/review/turn/retry, resume, packet cache, and routine paths bind or revalidate profile Skill bytes before provider launch. Prompt framing is subordinate and grants no unlisted script route, referenced file, asset, integration, network, or tool; model-authored workflows cannot select Skills.
- Fresh sessions receive the complete bound profile section. Same-session direct recovery and resumed team plan/review/turn/quality-retry calls re-read and verify the complete section, then use a hash-and-byte-count continuation receipt only when it is smaller. The receipt omits raw profile/context/Skill/memory text, grants no authority, and remains bound to the exact existing session; local profile inspection reports deterministic full/resumed/saved byte evidence with zero provider calls.
- Omitting profile `skills` performs no Skill lookup, read, prompt branch, cache field, state metric, or command work.
- An opt-in isolated profile tool policy uses minimal read/write workspace permissions, no network or native web, ignored user config/project rules, sanitized shell environment, bounded tool output, and disabled hook/plugin/app/browser/computer/memory/dependency/nested-agent surfaces. Omitted policy remains standard for compatibility; a tool-output ceiling on standard is invalid.
- A restricted profile retains the isolated envelope but enables one hash-bound session `PreToolUse` hook. It accepts only structured exact or prefix Bash argument rules plus optional `apply_patch`, rejects compound/expanding shell syntax, disables alternate execution features, uses a non-login shell, pins the compatible Codex CLI range, disables unmanaged external hooks after a two-pass audit, retains managed administrator hooks, and records policy evidence. Prefix rules grant trailing-argument semantics; unsupported native-tool paths are outside the hook guarantee and are never described as universally allowlisted.
- Profiles may opt into strict workspace-scoped external memory. Stores are revisioned, redacted, no-follow, size-bounded, atomically replaced under a writer lock, and never created in the repository by default. Snapshot selection defaults to legacy newest-first `recent-v1`; `lexical-relevance-v1` remains compatible presence-based rarity/tag ranking. Opt-in `hybrid-lexical-v1` adds bounded frequency, adjacent phrase, exact structured-identifier, tag, rarity, and integer length signals. Both query-based policies derive the same capped immutable task query, use newest-first ties, and restore chronological prompt order. Each run receives an immutable untrusted snapshot whose revision/hash and selector/query identity enter prompt, fingerprint, cache, trace, state, and dashboard evidence; resume restores that snapshot while later runs see newer revisions.
- Operator memory lifecycle commands are explicit local reads or writes. A workflow `agent_memory` append additionally requires a read-write profile, medium/high risk, write capability, and `agent-memory-write`; default entry IDs make interrupted commit replay exactly-once. Model-authored workflows cannot opt into memory, and `run-goal` rejects memory updates before external verification.
- Isolated profile context rejects `.env`, `.env.*`, PEM, and key paths. Tool-policy or output-limit changes invalidate workflow resume and agent-map cache identity before provider launch.
- Profile context is regular symlink-free bounded UTF-8. Ordered path/size/SHA-256 records enter the persisted workflow fingerprint and raw context stays out of lifecycle metadata; context drift fails before provider launch and invalidates resume, later goal iterations, and routine authorization.
- Generated and bundled agent workflows use balanced defaults: `high`/12,000 for ordinary calls, 24,000 for write calls, and `medium`/4,000 per map packet with 100,000 per map attempt.
- `balanced-v2` maps additionally cap model packets at 25. Source items are individually validated, deterministic grouped scopes contain at most 64 items, and impossible dynamic grouping fails before any provider launch.
- A map total is divided over uncached packets only. Cached packets receive no new allocation, and an underfunded fair share fails before any provider launch.
- A retry is a new bounded map attempt. Hard local loss may leave prior provider spend uncertain, so configured caps must not be reported as recovered or measured usage.
- Grouped cache identity binds every member and bounded content fingerprint. Any changed or uncacheable member invalidates its packet; targeted retry binds and reruns the complete canonical grouped scope while preserving compatible siblings.
- For substantial runtime-backed work, serial `auto` compiles one direct worker without a planning call. Omitted-cap read-only direct workers receive 4k/8k/12k/16k/20k by resolved low/medium/high/xhigh/ultra effort; direct writes receive 8k/16k/24k/32k/40k, and explicit caps remain authoritative. Diagnosis/review prompts group cross-method manifestations of one violated invariant, merge candidates sharing one first faulty executable line or repair site, retain one missing validation boundary as one finding across clause types, and move secondary symptoms to evidence. Deterministic Ultracode synthesis uses the same rule. A verifier plus explicit multi-worker envelope selects progressive adaptation: one direct attempt first, then same-session model planning only after verifier failure. `direct`, `progressive`, `workflow`, and `goal` strategies pin direct-only, direct-first escalation, one-shot decomposition, and plan-first adaptation respectively. Explicit `native` compiles one read-only parent with one funded depth-1 child by default, zero planner calls, dual-version thread enforcement, pruned external surfaces, and a deterministic-v5 `24k + 18k * child_slots` shared cap. `parent-integration-and-synthesis-v4` replaces duplicate generic checkpoint prose, adds one invariant/output-contract synthesis pass, and consolidates methods or call sites in one component that bypass the same required helper into one component-level root cause while preserving separately repaired components. Explicit direct benchmarks may use the retained `12k + 6k * child_slots` minimum, and deterministic-v4 receipts keep their original cap. Complete aggregate reconciliation fails a measured overage even when the provider returned success; unavailable reconciliation remains advisory. Task text controls neither topology nor capabilities.
- Native execution requires agent, parallel, dedicated delegation, and execution authority. Plan-only uses zero provider calls. Writes, adaptive verifiers, maps, isolated/restricted profiles, recursive fan-out, and model-authored self-grant fail before provider launch. Current routing/progress receipts expose runner-owned limits while nested lifecycle and usage attribution remain explicitly incomplete.
- Omitted auto effort uses `low` only for direct/progressive work with an explicit verifier, at most 400 task characters, and either one smallest-change public-function repair or one deterministic named-file test addition with no production change. Other verified direct/progressive work of at most 1,000 characters may resolve to `medium` when no broad, sensitive, or independently hard marker remains; everything else resolves to `high`. Current v6 keeps migrations, refactors, idempotency work, explicit multi-file/multi-module scope, and the existing repository-wide, security, schema, concurrent, transactional, production, and broad-test markers out of `low`, while frozen v1-v5 decisions remain reproducible. Explicit `low|medium|high|xhigh|ultra` is unchanged, and receipts store the resolved provider effort.
- Missing `model-workflow-execute` turns a one-shot auto route into compile/preflight only; serial direct plan-only launches zero provider processes. Adaptive or background routing fails before provider launch without execution approval; write execution still requires both `--allow-writes` and `model-workflow-write`.
- An explicit missing command host is a transport failure. The skill does not repeat the same failed route or misclassify it as repository evidence. `doctor` detects missing, broken, non-executable, and release-mismatched Codex companion hosts; approved repair creates only a missing sibling link under the user home and never replaces an existing path.
- Direct attempts remain single-worker and use existing weighted rollout caps and workflow/run receipts. The worker treats the runtime as the complete workflow and never reopens skills/plugins/goals. External verification replaces internal test/diff loops; auto-selected low/medium writes receive one combined pre-edit read, high compact work receives at most two, and low/medium/high workers prefer one patch call for two to eight required files while preserving failed/incomplete-write correction. `xhigh`/`ultra` remain unrestricted. Progressive resume never replays a completed direct attempt, detached payloads bind escalation mode, and goal/static/live views expose per-iteration planning mode plus completed planner-call totals. Legacy deterministic-v1/v2/v3 routing remains readable but cannot claim current v4 native semantics.
- A terminal failed write workflow with a real run directory may satisfy a command-only model goal when the retained workspace passes its deterministic verifier. The iteration retains `run_status: failed`; missing-run, blocked, interrupted, read-only, and model-verifier failures remain fail-closed.
- Goal routes accept command-only, model-only, or explicit `command+model` verification. Composite mode always runs the command first; failure or timeout short-circuits with zero model-verifier calls, while command success permits one semantic verdict. Every semantic verifier uses explicit high effort, an 8,000 weighted-token runtime-hard cap, an exact 4,000-token completion checkpoint, the isolated host-independent read-only profile, and an ephemeral session. Resume, auto receipts, and detached payloads bind both components and their order. Task text cannot select this topology.
- Composite model cutoff, launch, telemetry, timeout, and malformed-verdict errors fail closed as `verifier_failed`; an ordinary unsatisfied semantic verdict remains bounded retry feedback. Goal state and static/live detail expose verifier effort, cap, enforcement, isolation, ephemeral-session use, policy version, both component statuses, skips, and exact semantic-verifier calls without criteria text.
- One exact legacy model/composite verifier fingerprint for the same prompt/model/command may resume only by tightening to the current fixed policy. The resume records the policy migration and rewrites the current fingerprint/metadata before execution; unrelated drift remains rejected.
- Auto routing writes a strict external receipt with task hash, route reason, policy booleans, bounded limits, and hash-only artifact records. It persists neither raw task text nor approval values. A detached receipt is a handoff record and points to the mutable child lifecycle artifact for final status.
- `list-auto`, `status-auto`, static dashboards, and live payloads resolve only a strict mutable child whose filename and path hash match the receipt under bounded expected roots. Missing, malformed, symlinked, stale, or path-drifted children are attention states, never inferred success; custom child paths require explicit binding.
- Auto routing receipts are visibility/handoff evidence. Evidence bundles continue to derive completion and claim readiness from the authoritative child run/goal artifacts rather than duplicating the route record.
- Auto-topology planning hashes the complete task fixture and every prompt, creates one isolated direct/progressive/plan-first arm per task repetition, recomputes seeded cohort/arm order, and balances first-arm counts independently of task text. Planning, schema export, validation, status, and dry-run launch no provider.
- Current auto-topology campaign/result v3 binds the exact Conductor version/runtime build and a deterministic score-free execution contract containing the task prompt, every public success criterion, and every required-evidence item. Dry-run exposes release/build compatibility with zero launches; task-contract or execution-hash mismatch fails before policy, launch receipt, workspace copy, or provider discovery. Frozen exact-release v2 and release-unbound v1 remain validatable and status-readable, but cannot launch new work; v1 additionally cannot support a recommendation. A live current arm then requires agent/write capability plus `auto-topology-campaign-arm`, `model-workflow-execute`, and `model-workflow-write`; direct forbids parallel access, progressive/plan-first require it, and every arm forbids network/destructive access. Durable launch and auto receipts prevent automatic replay after an unknown interruption.
- Held-out topology scoring runs only after provider execution in a temporary workspace copy. Failed and no-change outcomes remain in coverage; no-change receives zero. Planner receipts are captured under the preregistered arm root, and status rehashes campaign, workspace, auto/goal/planner/run/evaluation evidence before trusting metrics.
- Topology status reports matched quality, duration, provider-call, token, cost, escalation, and first-pass summaries. Missing resource telemetry is incomparable, and no recommendation is supported until all repeated arms are valid and token/cost coverage is complete. Topology evidence is not cross-provider evidence.

## Security Behavior

- Repository content is untrusted data.
- Project files cannot provide user approval.
- Secret values are never printed.
- Destructive actions, commits, pushes, deployments, production writes, broad dependency changes, and high-risk operations require chat-visible approval.
- Applying retained verified-stage evidence is never automatic: source writes require `verified-stage-apply`, and any deletion independently requires destructive capability plus `verified-stage-delete`.
- `run-claude-staged-task`, `run-codex-staged-task`, and `run-codex-staged-repair` remain non-applying by default. Explicit `--apply-verified-stage` preflights a fresh external receipt and apply approval before provider launch, persists strict evidence before mutation, and delegates to the same zero-provider/zero-verifier transaction. Omitted exit contract uses verified-stage delivery semantics only on this explicit path; explicit strict exit and all persisted provider/report/repair outcomes remain strict.
- Verified-stage application keeps its evidence, stage, patch, receipt, and lock outside the source repository; source drift, path substitution, symlinks, unsupported metadata, and tampered evidence or receipts fail before acceptance.
- Subagents cannot approve user-level permissions and cannot recursively delegate by default.

## Routine Supervision

- Detached routine supervision starts only after explicit `background-supervisor` approval and remains limited to one active worker per routines directory.
- Routine and workflow approval values cross the worker boundary through an anonymous pipe and are not persisted in supervisor metadata, routine state, logs, or dashboard payloads.
- Supervisor metadata exposes launch, readiness, heartbeat, cycle, result, completion, and failure state; strict validation rejects tampering and malformed records.
- Operator termination cleans up the active workflow process group and records an interrupted routine instead of leaving it marked as running.
- Routine definitions support manual, interval, daily, weekdays, weekly, monthly, one-shot, and strict five-field numeric cron schedules. Cron supports wildcards, values, lists, ascending ranges, and positive steps; Sunday aliases and Vixie day-field behavior, including stepped wildcard flags, are deterministic, while names, extensions, macros, seconds, and malformed syntax fail closed. Monthly days clamp to the final day of short months. Restart catch-up selects at most the latest eligible slot from the preceding seven days, never replays a consumed slot, skips nonexistent DST minutes, and can consume each repeated local minute once.
- Pause suppresses and records each due slot once; run-now remains available while paused; resume and schedule updates require the routine approval.
- A routine written from an installed or plugin `.pyz` self-launches that absolute runtime outside a source checkout and binds its SHA-256. Runtime drift, another target, path drift, worker-ceiling drift, or a command capability that disagrees with manifest policy fails before workflow launch; legacy source-module manifests remain valid.
- Control events and execution history are strict, bounded, external to the repository, visible in CLI/static/live surfaces, and contain no approval values.
- Detached supervision is while-awake local execution. It does not install a daemon or restart after reboot; bounded catch-up begins only after an operator starts the supervisor again.
- Restart persistence requires the separate `install-routine-service` path. It must bind an expiring grant to exact packaged runtime bytes, listed routine manifests, target fingerprints, capabilities, and hashed workflow approvals; reject `all`; execute no unlisted routine; recheck the on-disk grant each cycle; and surface descriptor, state, heartbeat, PID, grant, runtime, or manifest drift as attention. Skill/runtime installation alone never enables it.
- Routine target fingerprints include referenced agent-profile context and complete selected Skill-tree hashes. Changing either resource after manifest or grant creation requires a new reviewed routine manifest and cannot silently alter a scheduled agent prompt or on-demand resource.

## Evidence Behavior

- Final answers separate observed evidence, inferred conclusions, and unverified assumptions.
- Checks are only reported as passed when actually run and observed.
- A verified-stage apply receipt advances durably through `prepared`, `merging`, and `merged`; interrupted file application reconciles only exact before/verified values and performs no provider or verifier replay.
- A terminal verified-stage receipt may be rechecked without approval or writes only when the current source still matches the recorded final fingerprint.
- Skipped or unavailable checks are named directly with the reason.
- Remaining risks and useful next actions are stated without overstating certainty.

## Agent-Map Recovery

- `retry-packet` accepts only an inactive run, an `agent_map` step, and a canonical packet index already present in bounded trace evidence.
- A targeted retry is bound to the current item SHA-256, increments only that packet's generation, marks the parent step pending, and preserves compatible sibling cache entries.
- `--cascade` resets transitive downstream steps without invalidating sibling packets in the selected upstream map.
- Before targeted execution, the prior regular output is removed through no-follow-safe IO so a failed rerun cannot inherit stale success evidence.
- Item drift, malformed trace rows or duplicate JSON keys, unsafe historical item sources, out-of-range or unobserved indices, and tampered generation state fail before provider launch.
- Whole-step reset or restart clears targeted packet generations and remains the fallback for historical packet evidence that cannot be bound safely.
- CLI, diagnostics, static detail, and live controls expose the selected packet and retry generation. The control edits state only; provider work begins on a later gated `run --resume`.
- Each successful packet writes a strict atomic cache receipt before returning. Resume may rebuild missing aggregate cache entries from valid receipts only after rechecking the complete cache fingerprint and output hash.
- Each cacheable read-only packet has one bounded baseline terminal observer. One stable `thread.started` and exactly one final `turn.completed` may write strict private `conductor.agent_map_packet_terminal.v1` before process return only when output, deterministic redaction, prompt/cache identity, launch-time budget, and every source item are exact. Recovery runs before receipt/cache lookup or provider launch, promotes through the ordinary receipt, and imports telemetry exactly once by terminal hash.
- Public packet-terminal validation, inspection, schema, static/live/API views, and generic artifact listings expose only bounded identity, counts, totals, and hashes. Raw session, provider events, path, and body remain private. Artifact/output/source/state/hash drift fails closed with zero provider calls.
- `retry-step` may recover pending packet terminals. Targeted retry, reset, skip, cascade invalidation, and restart reject pending local completions before changing state. Missing events, uncacheable source identity, unstable output, in-flight work, and remote completion lost before exact local evidence retain the explicit uncertain-retry boundary.

## Interrupted Runs

- One execute-wide no-follow advisory lock owns each run. A concurrent resume fails before heartbeat or state ownership changes.
- Runner metadata records bounded active child PID, process-group, session, and start time without argv, prompt, command text, or approval values.
- Handled exceptions finalize a still-running step as failed after active children unwind. A `running` step left by hard loss cannot be resumed implicitly.
- Every ordinary `codex_exec` has one bounded baseline terminal observer. One stable UUID-bearing `thread.started` followed by exactly one final `turn.completed` may write strict private current `conductor.codex_step_terminal.v2` before process return only after the canonical raw and deterministic redacted output checkpoints exist. V2 additionally binds the exact invocation base/effective prompt and any lifecycle-context receipt, including same-thread resume context; retained v1 remains readable. Direct-write calls additionally bind the exact terminal workspace result and resolved workspace-path hash with one terminal-time snapshot and no prelaunch scan.
- Ordinary retry/resume consumes valid single-step terminal evidence before continuation or fresh provider launch, accepts only raw or redacted output checkpoints, verifies the active or derived completed Codex checkpoint and direct-write workspace, imports telemetry exactly once by terminal hash, and completes the step with zero provider calls. Crashes after telemetry, checkpoint, or step commit remain idempotent.
- Single-step terminal validation, inspection, schema, diagnostics, and static/live/API views are hash-only. Generic artifact views suppress the private body; raw session, provider events, output path, and direct-write file identity are not public. Artifact, output, workspace, checkpoint, prompt, model, effort, budget, sandbox, workflow, or workspace-path drift fails before provider launch.
- Reset, skip, restart, and affected cascade controls reject pending locally completed Codex turns before state mutation. Missing events or unstable evidence preserve ordinary post-return validation or the explicit uncertain-turn recovery path and add no model call, prompt branch, permission, token reservation, network access, daemon, or background process.
- `recover-run` requires a free cross-process lock, same-host metadata, an inactive owner, strict child-process evidence, and no live recorded child identity.
- Recovery marks an uncertain running step failed by default. `--retry-running` explicitly resets only that step to pending; interruption between steps preserves completed work and derives the next resumable or terminal state.
- A single interrupted `codex_exec` with a strict active checkpoint may instead use mutually exclusive `--resume-codex`. Recovery binds the exact checkpoint hash, and the next normal resume starts a new turn on the same UUID-bound Codex thread without replaying the original prompt.
- Codex continuation rejects missing, terminal, malformed, exhausted, configuration-drifted, identity-drifted, or post-approval-tampered checkpoints before provider launch. It never claims to recover the exact in-flight turn or its uncertain token spend.
- Direct retry/reset/skip/packet controls cannot race active or interrupted execution. Pause, stop, and restart remain boundary requests, while terminate remains the validated OS signal path.
- Recovery finalizes runner metadata, appends a bounded control event and decision-log row, and never claims to continue an unproven in-flight provider call.

## Provider Parity Evidence

- Campaign planning requires an explicit seed, hashes every task contract, covers every task/repetition exactly once, randomizes trial order, and balances provider-first assignment within one row.
